Title: Roblox And Rojo Resolution

Assigned to: Agent 1
Lane: Domain
Status: done
Objective: Make CodeNexus genuinely useful on Rojo-based Roblox projects by layering Roblox- and Rojo-aware resolution on top of the now-working Luau language support, so imports, module relationships, and code navigation reflect Roblox project structure instead of only plain filesystem-level Luau semantics. This epic should ground itself in official Rojo project rules, use Rojo sourcemap output as a correctness oracle where useful, and validate against both small fixture repos and at least one real Rojo-style project.
In scope: Rojo project discovery; `default.project.json` parsing; filesystem-to-DataModel path mapping; a dedicated Roblox/Rojo resolution seam in the indexing layer; Roblox-aware resolution of static patterns such as `game:GetService(...)`, `script.Parent`, and chained `WaitForChild(...)`; simple alias tracking for those patterns; Roblox-aware import resolution for `require(...)`; client/shared/server boundary modeling for Rojo-mapped repos; tests, fixtures, docs, and real-product proof on Rojo-style projects.
Out of scope: non-Rojo Roblox projects; project-file formats beyond `default.project.json` unless explicitly added later by contract update; live Studio inspection; world projection ingestion; dynamic DataModel mutation tracking; full type-aware Roblox API semantics; branch/index redesign; automatic refresh; general Lua/Luau parser redesign; broad data-flow analysis for arbitrary alias propagation.
Dependencies: `/Users/alex/Projects/GitNexusFork-agent-1/planning/master-intent.md`; `/Users/alex/Projects/GitNexusFork-agent-1/planning/epics-todo.md`; `/Users/alex/Projects/GitNexusFork-agent-1/docs/policy.md`; `/Users/alex/Projects/GitNexusFork-agent-1/docs/architecture/luau-core-support.md`; `/Users/alex/Projects/GitNexusFork-agent-1/docs/architecture/repo-local-implementation.md`; Luau support implemented in `/Users/alex/Projects/GitNexusFork-agent-1/gitnexus/src/core/ingestion/*`; current import and call seams in `/Users/alex/Projects/GitNexusFork-agent-1/gitnexus/src/core/ingestion/import-processor.ts`, `/Users/alex/Projects/GitNexusFork-agent-1/gitnexus/src/core/ingestion/workers/parse-worker.ts`, and `/Users/alex/Projects/GitNexusFork-agent-1/gitnexus/src/core/ingestion/call-processor.ts`; real product/runtime surfaces already in place (`codenexus init/index/status/serve`); official Rojo references including [project format](https://rojo.space/docs/v7/project-format/) and [sync details](https://rojo.space/docs/v7/sync-details/); `luau-lsp` as a reference-only comparison source ([repo](https://github.com/JohnnyMorganz/luau-lsp)); primary real-world validation target `/Users/alex/Projects/roblox/dancegame-agent-1`; expected implementation home in a dedicated Roblox/Rojo-specific helper or module rather than scattering logic across generic Luau support.
Risks: if Epic 08 guesses at Roblox runtime structure instead of grounding itself in official Rojo project rules and project mappings, it will create misleading graphs; if it ignores Rojo sourcemap behavior as a comparison oracle, mapping mistakes may look plausible while still being wrong; if it tries to solve arbitrary dynamic Roblox patterns or broad alias/data-flow propagation, it will sprawl and become unreliable; if it mixes Roblox specialization into generic Luau support too deeply, later maintenance will get messy; if Rojo scope is not enforced, the epic can balloon into “support all Roblox repos” without the static guarantees needed for correctness.
Rollback strategy: keep Rojo-first as a hard boundary; if a Roblox pattern cannot be resolved statically through Rojo/project mapping plus simple syntax analysis, leave it unresolved explicitly rather than guessing; if a resolution heuristic proves too broad, narrow it and document the limit instead of preserving overreach.
