Title: Index Freshness And Manual Refresh Semantics

Assigned to: Agent 1
Lane: Runtime
Status: doing
Objective: Lock and implement the explicit freshness and manual-refresh behavior for CodeNexus now that the CLI and repo-local HTTP service are real, so `codenexus index`, `codenexus status`, and the live service all report and react to stale/current index state consistently.
In scope: defining and implementing the operational contract for stale versus current index state after repo changes; tightening how `codenexus index`, `codenexus status`, and `codenexus serve` interact with on-disk freshness; explicit manual-refresh behavior; live-service reporting when the on-disk index changes after startup; stale-state detail flags and operator-facing messaging; tests and docs for those behaviors.
Out of scope: automatic refresh, delta indexing, filesystem watchers, branch-aware indexing, separate per-branch indexes, Luau/Roblox support, world projection, CLI renaming, or transport/runtime redesign.

Risks: if this epic leaves freshness semantics half-implicit, `status` and the running service will drift in confusing ways; if it tries to solve automatic refresh now, it will sprawl into a different epic; if it overcouples live-service state to on-disk index rewrites, it may accidentally promise hot-reload behavior the product still does not support; if stale behavior stays too weak, users and agents will not know when the service is serving an outdated index; if the live service does not expose enough index identity to compare its loaded state with on-disk state, `serving_current` versus `serving_stale` will remain guesswork; if the loaded-index identity is not persisted consistently in both health and advisory runtime metadata, crash recovery and stale-runtime interpretation will drift.
Rollback strategy: keep the model explicit and conservative; if a freshness behavior proves wrong, revise the contract and implementation together rather than adding hidden heuristics; if a live-service edge case is unclear, prefer degraded/stale reporting over pretending the runtime is current; if a serving-state distinction cannot be proven from service identity plus loaded-index metadata, report the service as degraded rather than inferring freshness; if runtime metadata and health disagree about loaded index identity, treat health as authoritative and runtime metadata as stale.
