Title: Product And Runtime Boundary
Assigned to: Agent 1
Lane: Architecture
Status: doing
Objective: Lock the concrete product, runtime, and repo-state boundaries for CodeNexus so all later epics build against one explicit contract for repo ownership, command behavior, runtime state, and repo-local MCP service lifecycle.
In scope: the `.codenexus` ownership contract; config versus runtime-state separation; v1 command contracts for `cn init`, `cn index`, `cn status`, and `cn mcp serve`; repo-local MCP-over-HTTP runtime contract; repo lifecycle states; staleness/freshness product semantics; explicit v1 invariants and non-goals.
Out of scope: implementation changes to production code; multi-repo registry removal; CLI rewiring; MCP transport implementation; Luau parsing; Roblox/Rojo resolution; branch-aware indexing; embeddings; storage-engine migration; full `gitnexus` to `codenexus` rename sweep.
Dependencies: `planning/master-intent.md`; `planning/epics-todo.md`; `docs/policy.md`; `docs/README.md`; `docs/architecture/repo-local-product-shape.md`; current CLI/runtime reference points in `gitnexus/src/cli/analyze.ts`, `gitnexus/src/cli/mcp.ts`, `gitnexus/src/mcp/local/local-backend.ts`, `gitnexus/src/storage/repo-manager.ts`, and `gitnexus/src/server/mcp-http.ts`.
Risks: boundary definitions may stay too abstract to drive implementation; `.codenexus` ownership rules may omit failure-state recovery or stale-runtime cleanup; command contracts may conflict with existing CLI behavior in ways later epics discover too late; freshness semantics may be underspecified around missing index, dirty worktree, worktree identity, or branch changes; the command, runtime, and freshness sections may drift into contradictory state-transition rules if they are not locked as one coherent repo-state model; planning output may diverge from durable `docs/` output if canonical doc targets are not updated in the same pass; spec may accidentally freeze Kuzu-specific or TypeScript-specific details that should stay implementation-private.
Rollback strategy: keep this epic strictly at the contract/spec level; if later implementation reveals bad assumptions, revise the planning artifacts and the matching durable docs together before further code work rather than carrying conflicting behavior into downstream epics; if one contract section changes materially, update the companion invariants, state model, command/runtime docs, and evidence links in the same planning pass so downstream epics do not inherit split-brain behavior across `planning/` and `docs/`.

### [x] 101 Lock `.codenexus` Filesystem Layout and State Ownership

Surface: `.codenexus` layout contract; config-state contract; runtime-state contract; index-storage contract

The layout must stay minimal until future features actually require expansion: reserve no speculative paths for deferred features, and only pre-create the config file plus the minimum clearly-needed directories at init time. ### [x] 102 Lock V1 Command Contracts for `cn init`, `cn index`, `cn status`, and `cn mcp serve`

Surface: command contract for `cn init`; command contract for `cn index`; command contract for `cn status`; command contract for `cn mcp serve`; current behavior reference in `gitnexus/src/cli/analyze.ts` and `gitnexus/src/cli/mcp.ts`
Work: Lock the user-visible contract for each v1 command, including required preconditions, side effects, outputs, failure conditions, and exact repo-state transitions. The contract must state how each command behaves when run outside a repo, in an uninitialized repo, in an initialized but unindexed repo, in a stale repo, and in a repo with conflicting runtime state. It must also lock that `cn init` creates the chosen TOML config format and only the minimum clearly-needed directory structure, that `cn init` is idempotent and must not silently replace an existing configured port or other user-set config without explicit user action, that `cn index` is both the initial build and the manual refresh command in v1, and that `cn status` remains useful without a running service while being allowed to probe the service when it exists. `cn status` must remain read-only and must not mutate config, runtime state, or index state as part of probing. The final command contract must be deterministic enough that a caller can infer whether a failure is due to wrong repo context, missing initialization, missing index, stale state, or service/runtime conflict without consulting implementation details. Done when the command surface is specific enough that CLI implementation can be treated as mechanical execution of this contract rather than product design.
Tests: Validate the command contracts against representative success and failure paths including repeat init, idempotent init that preserves existing config, index without init, index used as manual refresh, serve without index, status on missing runtime state, status with stale runtime files but healthy live service, status with no live service and stale runtime files, nested-repo invocation that resolves to the nearest git root, and serve with configured-port conflicts. Validate that status probing is bounded and read-only at the contract level.
Docs: Update `planning/master-intent.md` and create or update `docs/cli/commands.md` with the v1 command contracts and state-transition expectations.

Done when downstream implementation epics can build the service without inventing lifecycle semantics on the fly.
Tests: Validate the runtime contract against start, stop, restart, stale runtime metadata, missing index, configured port already in use, duplicate serve attempts from the same repo boundary, conflicting worktrees configured to the same port, two repos serving simultaneously on different ports, and stale on-disk runtime files that disagree with live process state.

### [x] 104 Lock Freshness, Staleness, and Repo-State Semantics

Surface: freshness contract; staleness contract; repo-state model; current indexing/state reference in `gitnexus/src/cli/analyze.ts` and `gitnexus/src/storage/repo-manager.ts`
Work: Define the v1 semantics for current versus stale index state and the repo-state model exposed by CLI and MCP. The contract must state what counts as stale, when the system warns versus refuses work, what `cn status` must report, how manual refresh is expected to work, and how missing index, dirty working tree, changed commit, and branch change are represented at the product level even if branch-aware indexing is deferred. The locked v1 behavior must allow serving on stale indexes with explicit degraded reporting rather than hard refusal, must not silently auto-refresh on serve, and must specify how worktree-specific and branch-change staleness is surfaced without promising per-branch indexes yet. This unit must also choose and name one finite canonical repo-state set, produce one explicit repo-state matrix covering the transitions between those states, and specify which CLI/MCP surfaces are expected to warn, degrade, or fail in each state. `cn status` must report live service truth as authoritative when a service probe succeeds, and must report runtime metadata as stale when on-disk records disagree with live checks. Done when later epics have one stable answer for status and freshness behavior across CLI and MCP surfaces, and every command/runtime outcome maps back to one named canonical state.
Tests: Validate the state model against cases covering indexed-clean repo, indexed-dirty repo, changed HEAD, missing index files, stale service state, repo reopened after branch switch, separate worktrees with different freshness states, contradictory state inputs where on-disk metadata and live runtime checks disagree, live service with missing or stale runtime files, and stale runtime files with no live service. Validate that every command/runtime outcome maps back to one named canonical repo state.
Docs: Update `planning/master-intent.md` and create or update `docs/architecture/repo-state-model.md` with the canonical repo-state names, matrix, transition rules, and live-versus-advisory status precedence rules.
