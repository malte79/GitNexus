Title: Change Contracts For Agent Work Planning
Assigned to: Agent 1
Lane: Workflow
Status: doing
Objective: Add a bounded-confidence `change contract` capability to gnexus that helps planning, engineering, QA, and security agents scope work faster and more safely without claiming full codebase understanding. Success means gnexus can produce a codebase-specific contract for a requested change with explicit evidence buckets (`grounded`, `strong_inference`, `hypothesis`), exact likely edit surfaces, likely hidden dependencies, recommended tests, and risk notes, and can optionally verify a completed change against that contract. The feature is only successful if it improves real agent task outcomes on a pinned benchmark corpus: no decrease in final task success rate, at least a 20% median reduction in time to first correct edit on multi-file tasks, at least a 25% median reduction in orientation-token consumption, at least a 30% reduction in wrong-surface exploration, and no regression in QA/security test-target recall versus baseline.
In scope: a first-class CLI and repo-local MCP surface for generating bounded-confidence change contracts; a strict evidence model that distinguishes direct graph facts from structural inference and hypotheses; change-contract output that names exact candidate files, symbols, boundaries, tests, and risk surfaces; a post-change verification path that checks whether an implementation matches the contract and flags missed likely surfaces; a pinned benchmark corpus across planning, engineering, QA, and security tasks; measurement harnesses for time-to-first-correct-edit, total task time, token use, wrong-surface rate, rework rate, test-target quality, and final task success; acceptance thresholds and go/no-go criteria recorded in repo-owned docs and epic evidence.
Out of scope: pretending gnexus fully understands arbitrary codebases; silent overclaiming of certainty; a feature that emits one flat edit list without evidence classes; fully autonomous patch generation; productizing agent telemetry collection outside repo-owned benchmark runs; live hosted evaluation services; changing the storage engine, MCP transport, or repo-local state model beyond what the feature contract requires; broad prompt-engineering experiments without pinned benchmark comparability; shipping the feature if benchmark gains come from degraded correctness or inflated confidence.

Risks: the feature can become actively harmful if it overstates confidence or collapses evidence classes into one authoritative-sounding answer; benchmark gains can be faked by selecting easy tasks or by measuring only speed while correctness degrades; token measurements can be noisy unless the benchmark corpus, prompt shells, agent/runtime configuration, retry budget, and repo freshness state are pinned; freshness drift can silently change task difficulty if baseline and candidate runs do not prove identical indexed commit, serving state, and repo cleanliness before each run; orientation savings can be overstated or understated if the benchmark does not define one explicit orientation-phase boundary and force both baseline and candidate runs to start before any repo exploration occurs; “time to first correct edit” and wrong-surface rate are meaningless unless each task has an explicit gold adjudication source for correct edit surfaces, expected tests, and success criteria; implementation may sprawl into a generic planner instead of a structural contract generator; the verification path can become a second noisy heuristic layer if it does not stay grounded in contract fields and current graph evidence and distinguish implementation misses from contract insufficiency; broad task classes like QA and security can be underserved if the contract output is optimized only for code edits; agents may ignore the feature if the contract is verbose, repetitive, or not demonstrably better than existing `query`, `context`, and `impact` flows.
