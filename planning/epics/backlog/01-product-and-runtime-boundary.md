Title: Product And Runtime Boundary
Assigned to: unassigned
Lane: Workflow
Status: backlog
Objective: Lock the concrete product, runtime, and repo-state boundaries for CodeNexus so all later epics build against one explicit contract for repo ownership, command behavior, runtime state, and repo-local MCP service lifecycle.
In scope: the `.codenexus` ownership contract; config versus runtime-state separation; v1 command contracts for `cn init`, `cn index`, `cn status`, and `cn mcp serve`; repo-local MCP-over-HTTP runtime contract; repo lifecycle states; staleness/freshness product semantics; explicit v1 invariants and non-goals.
Out of scope: implementation changes to production code; multi-repo registry removal; CLI rewiring; MCP transport implementation; Luau parsing; Roblox/Rojo resolution; branch-aware indexing; embeddings; storage-engine migration; full `gitnexus` to `codenexus` rename sweep.
Dependencies: `planning/master-intent.md`; `planning/epics-todo.md`; current CLI/runtime reference points in `gitnexus/src/cli/analyze.ts`, `gitnexus/src/cli/mcp.ts`, `gitnexus/src/mcp/local/local-backend.ts`, `gitnexus/src/storage/repo-manager.ts`, and `gitnexus/src/server/mcp-http.ts`.
Risks: boundary definitions may stay too abstract to drive implementation; `.codenexus` ownership rules may omit failure-state recovery or stale-runtime cleanup; command contracts may conflict with existing CLI behavior in ways later epics discover too late; freshness semantics may be underspecified around missing index, dirty worktree, worktree identity, or branch changes; the command, runtime, and freshness sections may drift into contradictory state-transition rules if they are not locked as one coherent repo-state model; spec may accidentally freeze Kuzu-specific or TypeScript-specific details that should stay implementation-private.
Rollback strategy: keep this epic strictly at the contract/spec level; if later implementation reveals bad assumptions, revise the planning artifacts first and re-lock the contract before further code work rather than carrying conflicting behavior into downstream epics; if one contract section changes materially, update the companion invariants, state model, and command/runtime docs in the same planning pass so downstream epics do not inherit split-brain behavior.

### [ ] 100 Lock Product Identity, Repo Ownership, and Non-Negotiable Invariants

Surface: `planning/master-intent.md`; CodeNexus repo-ownership contract for `.codenexus`; command-surface contract for `cn`
Work: Lock the non-negotiable product invariants for CodeNexus so later epics cannot drift the product shape. This unit must define what CodeNexus is for, what it explicitly is not for, what state it may create inside a repo, what it must never mutate outside `.codenexus`, what “repo-local” means operationally, and what the single supported runtime audience and transport are in v1. The locked invariant set must explicitly treat the no-mutation rule outside `.codenexus` as absolute, define worktrees as separate runtime boundaries in v1, define nested repos as separate repo boundaries from their parents, require commands to resolve the nearest enclosing git root as the operative repo boundary, and state that config format is part of the runtime boundary rather than a deferred implementation detail. Done when the invariant set is explicit enough to judge future proposals as in-bounds or out-of-bounds without re-litigating product intent.
Tests: Validate the invariant set against concrete scenarios covering a clean repo, an already-initialized repo, a repo with only `.codenexus`, a repo with accidental external mutations, a repo opened through a separate worktree, a nested invocation from the wrong worktree, invocation from inside a nested repo under a parent repo checkout, and a repo with no running MCP service.
Docs: Update `planning/master-intent.md` with a dedicated invariants section and a locked v1 non-goals section.

### [ ] 101 Lock `.codenexus` Filesystem Layout and State Ownership

Surface: `.codenexus` layout contract; config-state contract; runtime-state contract; index-storage contract
Work: Define the v1 `.codenexus` filesystem layout and classify every path as config, derived index state, runtime state, or disposable temporary state. The layout must specify which files are created by `cn init`, which are created by `cn index`, which are created by `cn mcp serve`, what can be deleted safely, and what must survive across process restarts. The layout must stay minimal until future features actually require expansion: reserve no speculative paths for deferred features, and only pre-create the config file plus the minimum clearly-needed directories at init time. This unit must also lock the exact v1 config serialization format as TOML, the required config/runtime fields that later commands and services may rely on, and whether any command is allowed to rewrite an existing config file without explicit user intent. Done when each planned `.codenexus` path has a single purpose, ownership rule, lifecycle, and field-level contract that later epics can implement without inventing new state surfaces ad hoc.
Tests: Validate the layout against repo lifecycle cases covering first init, idempotent re-init, index rebuild, service crash and restart, stale runtime metadata, cleanup after deleting or recreating `.codenexus`, and absence of unused speculative directories after `cn init`. Validate that the config and runtime file contracts are explicit enough to catch missing required fields, stale advisory-only state, accidental format drift, and non-TOML config shape drift.
Docs: Add a `.codenexus` layout and TOML config/runtime file contract section to `planning/master-intent.md` or a dedicated planning-side contract doc referenced from it.

### [ ] 102 Lock V1 Command Contracts for `cn init`, `cn index`, `cn status`, and `cn mcp serve`

Surface: command contract for `cn init`; command contract for `cn index`; command contract for `cn status`; command contract for `cn mcp serve`; current behavior reference in `gitnexus/src/cli/analyze.ts` and `gitnexus/src/cli/mcp.ts`
Work: Lock the user-visible contract for each v1 command, including required preconditions, side effects, outputs, failure conditions, and exact repo-state transitions. The contract must state how each command behaves when run outside a repo, in an uninitialized repo, in an initialized but unindexed repo, in a stale repo, and in a repo with conflicting runtime state. It must also lock that `cn init` creates the chosen TOML config format and only the minimum clearly-needed directory structure, that `cn init` is idempotent and must not silently replace an existing configured port or other user-set config without explicit user action, that `cn index` is both the initial build and the manual refresh command in v1, and that `cn status` remains useful without a running service while being allowed to probe the service when it exists. `cn status` must remain read-only and must not mutate config, runtime state, or index state as part of probing. The final command contract must be deterministic enough that a caller can infer whether a failure is due to wrong repo context, missing initialization, missing index, stale state, or service/runtime conflict without consulting implementation details. Done when the command surface is specific enough that CLI implementation can be treated as mechanical execution of this contract rather than product design.
Tests: Validate the command contracts against representative success and failure paths including repeat init, idempotent init that preserves existing config, index without init, index used as manual refresh, serve without index, status on missing runtime state, status with stale runtime files but healthy live service, status with no live service and stale runtime files, nested-repo invocation that resolves to the nearest git root, and serve with configured-port conflicts. Validate that status probing is bounded and read-only at the contract level.
Docs: Add a v1 command-contract section to `planning/master-intent.md` and cross-link the command expectations from `planning/epics-todo.md` if needed.

### [ ] 103 Lock Repo-Local MCP HTTP Runtime Contract

Surface: repo-local MCP-over-HTTP contract; runtime metadata contract; service lifecycle contract; current HTTP/MCP reference in `gitnexus/src/server/mcp-http.ts`, `gitnexus/src/server/api.ts`, and `gitnexus/src/mcp/local/local-backend.ts`
Work: Lock the repo-local MCP-over-HTTP runtime contract for one repo, including how the service discovers repo config, binds to the configured port, records runtime state, reports health, and behaves when the repo is unindexed, stale, or already served. The contract must also state that runtime-state files are advisory only, that live process or port checks always win over stale on-disk runtime records, and that `cn mcp serve` is allowed to start on a stale index only if the service clearly reports degraded or stale status to clients. The runtime contract must explicitly define startup behavior for duplicate serve attempts from the same repo boundary and conflicting serve attempts from a different worktree bound to the same configured port. Same-repo duplicate serve attempts must fail loudly while reporting the existing healthy service details and must not rewrite runtime metadata unnecessarily, and different worktrees using the same configured port must fail loudly and require explicit port separation. Done when downstream implementation epics can build the service without inventing lifecycle semantics on the fly.
Tests: Validate the runtime contract against start, stop, restart, stale runtime metadata, missing index, configured port already in use, duplicate serve attempts from the same repo boundary, conflicting worktrees configured to the same port, two repos serving simultaneously on different ports, and stale on-disk runtime files that disagree with live process state.
Docs: Add a repo-local MCP runtime contract section to `planning/master-intent.md` or a dedicated planning-side runtime contract doc, including advisory runtime-state rules, duplicate-serve behavior, and port-conflict behavior across worktrees.

### [ ] 104 Lock Freshness, Staleness, and Repo-State Semantics

Surface: freshness contract; staleness contract; repo-state model; current indexing/state reference in `gitnexus/src/cli/analyze.ts` and `gitnexus/src/storage/repo-manager.ts`
Work: Define the v1 semantics for current versus stale index state and the repo-state model exposed by CLI and MCP. The contract must state what counts as stale, when the system warns versus refuses work, what `cn status` must report, how manual refresh is expected to work, and how missing index, dirty working tree, changed commit, and branch change are represented at the product level even if branch-aware indexing is deferred. The locked v1 behavior must allow serving on stale indexes with explicit degraded reporting rather than hard refusal, must not silently auto-refresh on serve, and must specify how worktree-specific and branch-change staleness is surfaced without promising per-branch indexes yet. This unit must also produce one explicit repo-state matrix covering a finite canonical state set, the transitions between those states, and which CLI/MCP surfaces are expected to warn, degrade, or fail in each state. `cn status` must report live service truth as authoritative when a service probe succeeds, and must report runtime metadata as stale when on-disk records disagree with live checks. Done when later epics have one stable answer for status and freshness behavior across CLI and MCP surfaces.
Tests: Validate the state model against cases covering indexed-clean repo, indexed-dirty repo, changed HEAD, missing index files, stale service state, repo reopened after branch switch, separate worktrees with different freshness states, contradictory state inputs where on-disk metadata and live runtime checks disagree, live service with missing or stale runtime files, and stale runtime files with no live service. Validate that every command/runtime outcome maps back to one named canonical repo state.
Docs: Add a freshness and repo-state contract section to `planning/master-intent.md`, including the explicit canonical repo-state matrix, transition rules, and live-versus-advisory status precedence rules.
