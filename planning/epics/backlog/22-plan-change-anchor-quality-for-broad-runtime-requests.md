Title: Plan-Change Anchor Quality For Broad Runtime Requests
Assigned to: unassigned
Lane: Workflow
Status: backlog
Objective: Improve `gnexus plan-change` so broad behavioral and runtime change requests anchor to the correct subsystem owners instead of isolated helpers, incidental geometry code, or playtest-only scenarios. Success is a preserved before-versus-after benchmark showing that broad requests choose the strongest subsystem cluster first, prefer owner-like anchors over leaf helpers when appropriate, surface a secondary coordinating anchor when one anchor is insufficient, and communicate ambiguity honestly when multiple clusters remain close, all without repo-specific heuristics.
In scope: universal anchor-quality improvements for broad `plan-change` requests; cluster-first anchoring across grounded surfaces; neighborhood and cohesion scoring across related candidates; stronger owner-vs-helper preference for broad behavioral goals; optional secondary coordinating anchor when a single anchor is misleading; explicit ambiguity or confidence signaling when top subsystem clusters remain close; a pinned benchmark corpus of broad runtime and cross-file change requests from multiple repos or synthetic fixtures that distinguishes good owner anchors from bad helper anchors; regression protection for exact or narrow owner lookups so the fix remains general-purpose and does not degrade existing targeted search behavior.
Out of scope: repo-specific path boosts, symbol boosts, helper blacklists, or prompt hacks; changes to `verify-change`; a new planning subsystem or parallel planner path; transport changes; benchmark sprawl outside anchor-quality and planning usefulness; widening `plan-change` into code generation or autonomous patch writing; silent fallback behavior that hides ambiguity instead of reporting it.
